 Analyse der PointClick2-Anwendung

  1. Unterschiede: SOLL vs. IST Filestruktur

  Im Schema definiert, aber NICHT vorhanden:
  - conditions_progress.txt (im Schema) → tatsächlich conditions-progress.txt (mit Bindestrich statt Unterstrich)

  Vorhanden, aber NICHT im Schema:
  - ✅ items-progress/ - Extra-Verzeichnis (wahrscheinlich für Runtime-Progress)
  - ✅ manual/ und manual2/ - Zusätzliche Verzeichnisse (vermutlich Entwickler-Notizen/Backups)
  - ✅ editor_settings.txt - Editor-Konfiguration
  - ✅ progress.txt - Spielfortschritt-Datei
  - ✅ images/itemsOrientation/ - Zusätzliches Bildverzeichnis für Item-Orientierungen
  - ✅ scenes/Beach/, scenes/Town/, scenes/Wood/ - SubScene-Unterverzeichnisse

  Schema-konform:
  - ✅ buttons/, conditions/, items/, dialogs/, processes/
  - ✅ images/{items, scenes, cursors}/

  3. Arrays und konkurrierende Strukturen in Item-Files

  Arrays in Item.java die ALLE auf dieselben .txt-Files zugreifen:

  A) LEGACY-SYSTEM (alt):
  - List<Point> clickAreaPoints (Zeile 41) - Einfache Punktliste

  B) NEUES SYSTEM (konkurrierend):
  - List<CustomClickArea> customClickAreas (Zeile 46)
  - List<MovingRange> movingRanges (Zeile 47)
  - List<Path> paths (Zeile 48)

  Alle drei Klassen (CustomClickArea, MovingRange, Path) enthalten:
  - List<Point> points - Polygon-Punkte

  Konkurrierende Booleans und Flags:

  4. File-Format-Details aus ItemLoader

  Das Programm unterstützt ZWEI Formate gleichzeitig:

  NEUES Format (Schema Simple):
  #CustomClickArea:
  -point1: x=590, y=429, z=1
  -point2: x=600, y=440, z=1

  #MovingRange:
  -point1: x=100, y=200, z=1

  #Path:
  -point1: x=50, y=50, z=1

  ALTES Format (Legacy):
  #CustomClickAreas:
  -CustomClickArea:
  --Points:
  ---point1: x=100, y=200, z=0

  #MovingRanges:
  -MovingRange:
  --KeyAreaName: someArea
  --Conditions:
  ---someCondition = true

  #Paths:
  -Path:
  --KeyAreaName: someArea

  Wichtig: ItemLoader.java konsolidiert Multiple Arrays in Ein Array pro Typ:
  - consolidatePointContainers() (Item.java:696-723)
  - Mergt alle CustomClickAreas → primary CustomClickArea
  - Mergt alle MovingRanges → primary MovingRange
  - Mergt alle Paths → primary Path

  Zusammenfassung der Probleme

  🔴 Kritische Konflikte:
  1. 3 konkurrierende Click-Area-Systeme gleichzeitig aktiv
  2. IsInInventory doppelt gespeichert (Field + Condition)
  3. Legacy + New Format parallel unterstützt
  4. Keine klare Trennung zwischen Schema-Struktur und Runtime-Struktur

  🟡  Design-Inkonsistenzen:
  - Schema zeigt singular (#CustomClickArea:) aber Code unterstützt auch plural (#CustomClickAreas:)
  - Items können in mehreren Arrays gleichzeitig sein (clickAreaPoints + customClickAreas)
  - MovingRange/Path erben von CustomClickArea (Code-Wiederverwendung vs. semantische Verwirrung)
